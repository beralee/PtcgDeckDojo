# 当前状态与已知限制

## 项目状态

`PtcgDeckDojo` 目前更适合作为“可运行、可继续开发的练牌引擎项目”，而不是已经封版的最终产品。

公开仓库时，建议把它描述为：

- 一个基于 Godot 的 PTCG 本地练牌项目
- 一个持续补卡、补交互、补测试的规则引擎仓库
- 一个适合继续做效果系统和 UI 演进的开发底座

## 当前已经比较明确的能力

- 主菜单、卡组管理、对战设置、战斗主界面已连通
- 可以从 `tcg.mik.moe` 导入卡组并缓存卡牌与卡图
- 规则引擎、效果注册和大量卡牌效果脚本已经存在
- 已有较多自动化测试覆盖数据层、效果层、部分 UI 和编码质量

## 当前不应夸大的部分

- AI 对战仍不应在 README 中宣传为已完成能力
- 设置页仍是占位状态
- 部分效果脚本仍有 `TODO` 注释，说明交互尚未完全补齐
- 阶段性文档中包含特定批次的补卡记录，不代表整个项目已全部完成

## 已知协作风险

### 1. 编码风险

项目历史上出现过中文编码污染问题，因此：

- 文本文件必须统一使用 UTF-8
- 不要把终端乱码直接写回源文件
- 批量重写文档或脚本后必须重新检查

### 2. 规则与 UI 脱节风险

很多表面上的“UI 问题”本质上是规则、状态流或交互协议问题。补功能时不要只在界面层打补丁。

### 3. 公开仓库素材边界

Pokemon 卡图和卡牌内容涉及第三方知识产权。当前更稳妥的做法是：

- 仓库不提交本地缓存资源
- 由用户运行时自行下载
- 在 README 中明确非官方和非商业声明

## 适合优先推进的事项

- 补齐带 `TODO` 的效果交互
- 收紧 README 与设计文档中的实现描述，避免超前宣传
- 持续补充语义回归测试
- 整理发布流程、版本号和 changelog

## 开源维护建议

- 对外文档保持“真实、保守、可验证”
- 每次合并规则或效果改动都附带验证结果
- 不要把批量审核门槛降到只为通过测试
