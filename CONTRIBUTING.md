# Contributing

感谢你考虑为 `PtcgDeckDojo` 提交改动。

这个项目目前仍在快速迭代，贡献前请先接受一个前提：这里更看重规则正确性、编码稳定性和回归可验证性，而不是“先跑起来再说”。

## 提交前先读

- [README.md](README.md)
- [docs/README.md](docs/README.md)
- [docs/development_setup.md](docs/development_setup.md)
- [DEVELOPMENT_SPEC.md](DEVELOPMENT_SPEC.md)

## 适合贡献的方向

- 补充缺失的卡牌效果实现
- 将现有 `TODO` 交互迁移到统一交互步骤框架
- 修复规则结算错误
- 补充自动化测试与回归用例
- 改善文档、测试入口和开发体验

## 基本原则

- 所有文本文件统一使用 UTF-8
- 不要把终端乱码或错误解码文本写回仓库
- 修改前先读现有实现，不凭记忆假设
- 涉及规则、效果、交互链路时，优先补测试
- 不要为了图省事绕开既有 `BaseEffect` 和交互框架

## 开发流程

推荐按下面的顺序工作：

1. 确认问题和改动范围
2. 阅读相关脚本、场景与设计文档
3. 先补测试或先明确验证方式
4. 实现改动
5. 运行测试与手动回归
6. 在 PR 描述中说明行为变化和验证结果

## 测试要求

以下改动默认应附带验证：

- 规则引擎
- 效果脚本
- 战斗主场景
- 卡组导入与缓存逻辑
- 文本批量修改或编码修复

常用验证方式：

- 运行 `res://tests/TestRunner.tscn`
- 针对卡牌逻辑运行批量审核脚本
- 手动验证战斗流程中的关键交互

## PR 建议写法

请尽量在 PR 中写清楚：

- 改了什么
- 为什么要改
- 风险点是什么
- 跑了哪些测试
- 是否涉及新的卡牌交互或资源依赖

## 不建议直接提交的内容

- `.godot/` 编辑器缓存
- 本地日志、临时文件、实验脚本输出
- 用户本地缓存的卡牌 JSON 与卡图
- 与仓库公开目标无关的 AI 助手配置

## 关于许可证与素材

如果你的改动涉及外部资源、卡图、第三方文本或模板，请在 PR 里明确来源与授权边界。对这类内容，宁可保守，也不要含糊。
